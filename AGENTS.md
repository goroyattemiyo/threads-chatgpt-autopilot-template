# ChatGPT／Codex 作業ルール

このリポジトリでは、ChatGPT／Codexは主にThreadsの投稿予定と投稿履歴を管理します。

## 基本方針

- 通常の依頼では、`posts/`と`config/service.yml`だけを編集する
- `src/`と`.github/workflows/`は、利用者が明示的に開発・修正を依頼した場合だけ変更する
- 投稿済み履歴を削除・書き換えない
- パスワード、アクセストークン、Personal Access Token、Secretsの値を尋ねない
- 認証情報をファイル、Issue、ログへ記録しない
- 対応SNSはThreadsのみとして扱う

## 投稿追加の手順

1. `config/service.yml`でタイムゾーンと投稿枠を確認する
2. `posts/schedules/`と`posts/posted_log.yml`から類似投稿・重複投稿を確認する
3. 対象日を含む週別YAMLを選ぶ
4. 一意の`id`を付ける
5. 日付、投稿枠、本文、状態を登録する
6. 画像付きの場合は`image_key`を登録し、URLがない間は`status: draft`にする
7. 変更したファイルと追加内容を利用者へ報告する

## 投稿状態

- `draft`：準備中
- `ready`：自動投稿可能
- `posted`：投稿済み
- `error`：投稿失敗
- `cancelled`：取り消し

利用者が投稿内容と日時を確定したと明確に判断できない場合は、`draft`で保存します。

## 自動投稿直前の安全確認

`status: ready`へ変更すると、設定時刻との関係によっては次回のGitHub Actions実行で投稿される可能性があります。

以下の場合は、勝手に`ready`へ変更せず`draft`で保存し、利用者へ確認します。

- 投稿時刻がすでに到来している
- 投稿時刻まで30分以内
- 日時または本文が曖昧
- 同じ投稿枠に別の`ready`投稿がある
- 類似投稿または重複投稿が見つかった
- 画像付き投稿で`image_url`が空
- ツリー内の画像URLが揃っていない

## 投稿済みデータ

- `status: posted`の項目は原則変更しない
- 誤字修正の依頼でも、実際のThreads投稿が変わらない限り履歴を改変しない
- 再利用する場合は新しい`id`の投稿予定として追加する
- 削除ではなく、必要に応じて新しい投稿との関係を管理項目へ記録する

## 日時変更

- `draft`または`ready`だけを変更対象とする
- `posted`は移動しない
- 変更先の日時に別の`ready`投稿がないか確認する
- 未定義の投稿枠が必要な場合は、利用者へ説明したうえで`config/service.yml`へ追加する

## キャンセル

投稿予定を消去せず、次のように変更します。

```yaml
status: cancelled
```

必要に応じて`cancelled_at`と`cancel_reason`を追記します。

## 画像付き投稿

- 画像ファイル名と`image_key`を一致させる
- 拡張子を`image_key`へ含めない
- `image_url`が空の状態で`ready`にしない
- 代替テキスト`alt`を可能な範囲で記入する
- Public画像リポジトリへ置く画像であることを利用者へ意識させる

## ツリー投稿

- 1件目をルートの`text`へ入れる
- 2件目以降を`thread_posts`へ順番どおり入れる
- 投稿順序を勝手に変更しない
- 各返信の画像URLが揃うまでルートを`ready`にしない

## 類似投稿の確認

少なくとも次を確認します。

- 同一またはほぼ同一の本文
- 同じテーマ・タイトル
- 同じ画像キー
- 直近に同内容が投稿済みでないか

類似候補が見つかった場合は、勝手に削除せず利用者へ候補を示します。

## 禁止事項

- Secretsの値を表示・推測する
- 顧客固有の認証情報をコミットする
- 投稿済みIDを空に戻して再投稿可能にする
- `status: posted`を理由なく`ready`へ戻す
- エラー原因を確認せず同じ投稿を再実行する
- Threads以外のSNS対応を、既存機能として案内する
- 利用者の明示依頼なしにワークフローやPythonコードを変更する
